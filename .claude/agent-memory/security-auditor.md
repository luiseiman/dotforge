# Security Auditor — Agent Memory

## 2026-04-05 — Full security audit of dotforge repository

- **Accepted risk:** `eval "$*"` in `global/sync.sh:192` (the `action()` function). All call sites pass hardcoded strings like `mkdir -p '...'`, `cp '...' '...'`, `remove_item '...' && install_item '...'`. Paths come from `basename`/`DOTFORGE_DIR`/`CLAUDE_HOME` — all derived from the filesystem, not user input. Not exploitable in practice from the sync workflow. If the repo itself were compromised, all bets are off regardless.
- **Accepted risk:** `eval echo "~${SUDO_USER}"` and `eval echo "~${repo_owner}"` in `global/sync.sh:136,146`. These expand tilde for a username. The values come from `$SUDO_USER` (set by sudo, not user-controlled in context) and `stat` output on the repo directory. Risk is low; an attacker would need to control the repo's filesystem owner.
- **Accepted risk:** `Bash(cat *)` allow rule in `.claude/settings.json`. In this repo (markdown + shell scripts only), `cat` on any file is not a secret-exposure risk. The deny rules for `Read(**/.env)` etc. only cover the `Read` tool, not `Bash(cat)`, creating a theoretical bypass. Accepted because this is a configuration-only repo with no secrets.
- **False positive:** `CLAUDE_KIT_DIR` in `stacks/hookify/settings.json.partial` — this is a stale variable name from before the rename to `dotforge`. The hookify stack hooks will silently fail if `CLAUDE_KIT_DIR` is not set. Not a security issue but a correctness bug.
- **False positive:** `Bash(/dev/null find:*)` in `.claude/settings.local.json` — this is an artifact/typo from a previous session. It allows commands matching the literal prefix `/dev/null find:` which no real command would ever match. Harmless malformed entry.
- **Watch:** `.claude/settings.local.json` contains `Bash(/usr/bin/env bash -c ':*)` — this allow rule effectively permits ANY bash command routed through `env bash -c`. This is the most significant finding in the repo. The entry is in `settings.local.json` (gitignored, local-only) but it bypasses the deny rules at the permission layer. Monitor that this does not get added to `settings.json` or `template/settings.json.tmpl`.
- **Watch:** `.claude/hooks/session-report.sh` has `DOMAIN_CHANGES` variable used at line 124 (in heredoc) but assigned at line 129. The JSON metric `domain_knowledge_updated` will always be `0` in the deployed hook (uses uninitialized variable which expands to empty, causing arithmetic to produce 0 or empty). The template version (`template/hooks/session-report.sh`) has this correct. The deployed `.claude/hooks/session-report.sh` is a diverged copy — needs sync from template.
- **Watch:** Deployed `.claude/hooks/block-destructive.sh` uses `rm -rf \*` pattern while template uses `rm -rf \.?\*`. The deployed pattern does NOT block `rm -rf .*` (dot-files). Template pattern covers both `*` and `.*`. Low risk in practice but a coverage gap.
- **Watch:** `audit/score.sh` Python heredoc embeds shell variables as Python triple-quoted strings. If any `n*` note variable ever contained `"""` the Python block would produce malformed code. Currently all note strings are hardcoded templates so this cannot be triggered, but the pattern is fragile.
